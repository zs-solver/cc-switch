# 配置可用性测试功能 - 开发记录

## 需求

对所有已注册的 API 配置进行批量可用性测试，获取以下指标：
- 状态（是否可用）
- 首字耗时（TTFT）
- 回复字数
- 总耗时
- 速度（字/秒）
- 命中的模型和路径
- 完整响应内容

## 实现方案

采用直接 HTTP 请求 `/v1/messages` 流式接口（方案 A），而非 `claude -p` 命令行方式。
理由：可精确测量 TTFT，所有配置可并发测试，数据干净可比。

## 文件结构

- `test_configs.py` — 测试核心逻辑，独立文件
  - `_test_url()` — 对单个 URL 发流式请求，解析 SSE 事件流
  - `test_single_config()` — 对一个配置并发尝试多种 模型×路径 组合
  - `TestAllDialog` — PyQt5 GUI 对话框（被 main.py 导入）
  - `__main__` — CLI 模式，控制台输出结果
- `main.py` — 托盘菜单新增「测试所有配置(&T)」入口

## 踩过的坑

### 1. 认证头
部分代理（如阿里云 DashScope）不认 `x-api-key`，需要 `Authorization: Bearer`。
解决：两个头同时发。

### 2. 模型解析
配置文件中模型可能在两个位置：
- 顶层 `model` 字段
- `env.ANTHROPIC_MODEL` 环境变量（如 GLM5 的 `glm-5`）

之前只读了顶层 `model`，漏掉 env 里的，导致用错误的默认模型请求，被代理拒绝。

解析优先级：`cfg.model` > `env.ANTHROPIC_MODEL` > FALLBACK_MODELS

### 3. 路径兼容
不同代理的 API 路径不统一，有的需要 `/v1/messages`，有的只需 `/messages`。
解决：每个配置同时测两条路径，取先成功的。

### 4. 多模型探测
除了有明确非标模型的配置（如 GLM5 的 `glm-5`），其余配置同时测试三种模型：
`claude-opus-4-6-thinking`、`claude-opus-4-6`、`opus`

每个配置最多并发 3模型 × 2路径 = 6 个请求。

### 5. GLM5 流式响应
GLM5 认证和非流式请求都正常，但流式 SSE 事件格式可能与标准 Anthropic 不同，
导致解析为"无响应内容"。待排查其 SSE 事件结构。
