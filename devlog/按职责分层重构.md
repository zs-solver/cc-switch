# 按职责分层重构 — 开发记录

## 背景

重构前项目只有两个源文件：

| 文件 | 行数 | 内容 |
|------|------|------|
| `main.py` | 484 | JSON 工具函数、配置领域逻辑、Qt 图标创建、AddConfigDialog、CCSwitch 托盘应用 |
| `test_configs.py` | 398 | 测试执行逻辑、测试配置加载、TestAllDialog GUI、CLI 入口 |
| **合计** | **882** | |

存在以下问题：

1. **全局可变状态**：`APP_CONFIG`、`CONFIGS`、`KNOWN_ENV_KEYS` 等模块级全局变量在模块加载时初始化，运行时被直接修改（`CONFIGS.extend(...)`、`APP_CONFIG["configs"] = CONFIGS`），状态散落在模块顶层
2. **逻辑重复**：`test_configs.py::load_test_configs()` 重新实现了 `main.py` 中的配置查找逻辑（路径搜索优先级、模型推导规则），两处逻辑需要同步维护
3. **职责混杂**：`main.py` 同时包含 `read_json()`/`write_json()` 通用工具、`find_cfg_path()`/`auto_discover_configs()` 配置领域逻辑、`create_icon()` Qt 图标创建、`AddConfigDialog` UI 类、`CCSwitch` 托盘应用类
4. **循环依赖风险**：`main.py` 导入 `test_configs.py`（`from test_configs import TestAllDialog`），如果测试面板需要调用 `switch_config()` 就会形成循环导入
5. **业务逻辑嵌入 UI**：`on_add_config()` 中的文件名 slug 生成、冲突检测、settings 文件写入、config.json 更新逻辑直接写在 UI handler 里，无法被其他入口（如 CLI）复用

## 重构目标

- 按职责拆分为 `core/`（纯逻辑，不依赖 Qt）和 `ui/`（Qt GUI）两层
- 消除循环依赖，建立严格单向依赖关系
- 所有配置状态收敛到单一 `ConfigManager` 实例，消除全局变量
- 统一重复的配置加载/模型推导逻辑
- 为后续功能（删除/编辑配置、测试面板右键菜单、自定义提问）建立可扩展的结构

## 重构后文件结构

```
cc-switch/
├── main.py                  # 入口 + CCSwitch 托盘应用（纯 UI 编排）
├── core/                    # 核心逻辑层（不依赖 Qt）
│   ├── __init__.py
│   ├── config_manager.py    # ConfigManager 类 — 配置全部状态与操作
│   └── test_runner.py       # 测试执行逻辑 + CLI 入口
├── ui/                      # Qt GUI 层（依赖 core/）
│   ├── __init__.py
│   ├── icons.py             # 图标创建工具函数
│   ├── dialogs.py           # AddConfigDialog（及未来的 Edit/Delete 对话框）
│   └── test_dialog.py       # TestAllDialog 测试面板
└── test_configs.py          # 已删除
```

## 依赖关系

```
core/config_manager.py   ← 零项目依赖，仅 stdlib
core/test_runner.py      ← 模块导入时零项目依赖（仅 stdlib + requests）
                           if __main__ 时导入 config_manager（同包）

ui/icons.py              ← 零项目依赖，仅 PyQt5
ui/dialogs.py            ← 零项目依赖，仅 PyQt5
ui/test_dialog.py        ← 导入 core.test_runner

main.py                  ← 导入 core.config_manager, ui.*
```

`ui/` → `core/` 单向依赖，`core/` 永远不导入 `ui/`，无环。

注意 `config_manager.py` 的 `get_test_list()` 方法内部通过延迟导入（`from core.test_runner import FALLBACK_MODELS`）引用了 `test_runner` 的常量，但这是同层内的单次引用，不构成架构层面的逆向依赖。

## 各模块改动详情

### 1. `core/config_manager.py`（221 行，新建）

**来源**：从 `main.py` 迁移

**模块级工具函数**：
- `read_json(path)` — 原 `main.py` 第 35-37 行，不变
- `write_json(path, data)` — 原 `main.py` 第 40-43 行，不变

**`ConfigManager` 类** — 将以下全局变量和独立函数收敛为实例属性和方法：

| 原全局变量/函数 | → ConfigManager 属性/方法 | 变化说明 |
|---|---|---|
| `SCRIPT_DIR` | `self.script_dir` | 默认值改为 `__file__` 上两级（`core/` → 项目根） |
| `CONFIG_FILE` | `self.config_file` | 由 `script_dir` 派生 |
| `LOCAL_SETTINGS_DIR` | `self.local_settings_dir` | 由 `script_dir` 派生 |
| `SETTINGS_DIR` | `self.settings_dir` | 从 `config.json` 读取后 `expanduser` |
| `SETTINGS_FILE` | `self.settings_file` | 由 `settings_dir` 派生 |
| `CONFIGS` (list) | `self.configs` (property) | 返回内部 `self._configs` 的直接引用 |
| `KNOWN_ENV_KEYS` (set) | `self.known_env_keys` | 同上 |
| `APP_CONFIG` (dict) | `self._raw` | 完整 config.json 内容，私有 |
| `load_config()` | `self._load()` | 构造函数自动调用 |
| `find_cfg_path(filename)` | `self.find_cfg_path(filename)` | 逻辑不变，引用实例路径 |
| `auto_discover_configs()` | `self.auto_discover()` | 逻辑不变，结尾调 `self.save()` 代替直接写 |
| `detect_current(settings)` | `self.detect_current()` | 不再需要外部传入 settings 参数，内部自己读 |
| `switch_config(filename)` | `self.switch(filename)` | 逻辑不变，引用实例路径 |
| `extract_website(base_url)` | `@staticmethod extract_website()` | 不变 |
| `derive_name(filename)` | `@staticmethod derive_name()` | 不变 |

**新增方法**：

- `save()` — 将 `self._configs` 回写到 `config.json`，替代原来散落各处的 `write_json(CONFIG_FILE, APP_CONFIG)`
- `add(base_url, api_key, name="", model="")` — 业务逻辑从 `CCSwitch.on_add_config()` 的非 UI 部分提取而来，包含：
  - 文件名 slug 生成（域名 → `settings-xxx.json`）
  - 冲突检测（同名加数字后缀）
  - settings 文件内容生成和写入
  - 追加到配置列表并持久化
  - 返回新创建的 entry dict
- `get_test_list()` — 合并了 `main.py::on_test_all()` 和 `test_configs.py::load_test_configs()` 中重复的逻辑，统一的测试配置列表构建

### 2. `core/test_runner.py`（185 行，从 `test_configs.py` 迁移）

**来源**：`test_configs.py` 的非 GUI 部分

**保持不变的内容**：
- `API_PATHS` 常量
- `FALLBACK_MODELS` 常量
- `_test_url(url, headers, body)` — 流式请求和 SSE 解析
- `test_single_config(name, base_url, api_key, models)` — HTTP 并发测试
- `test_single_config_cli(name, settings_path)` — CLI 测试

**`if __name__ == "__main__"` 块改动**：
- 新增 `sys.path` 修正（将项目根加入 path，解决 `python core/test_runner.py` 直接运行时的包导入问题）
- 用 `ConfigManager` 实例的 `get_test_list()` 替代原来的 `load_test_configs()`
- 其余逻辑（并发执行 HTTP + CLI 测试、格式化输出）完全不变

### 3. `ui/icons.py`（34 行，从 `main.py` 迁移）

**来源**：
- `create_tray_icon()` ← `main.py::create_icon()` 第 153-168 行（仅重命名）
- `create_check_icon()` ← `CCSwitch.create_check_icon()` 第 256-267 行（从实例方法变为模块函数）

### 4. `ui/dialogs.py`（74 行，从 `main.py` 迁移）

**来源**：`main.py::AddConfigDialog` 第 171-239 行

代码完全不变，仅移动到独立文件。`urlparse` 的导入随之迁移。

### 5. `ui/test_dialog.py`（185 行，从 `test_configs.py` 迁移）

**来源**：`test_configs.py` 的 GUI 部分（第 167-348 行）

**改动**：

| 项目 | 重构前 | 重构后 |
|------|--------|--------|
| Worker 类名 | `_TestWorker`、`_CLITestWorker` | `TestWorker`、`CLITestWorker`（去掉下划线前缀） |
| 函数导入 | 同文件直接调用 | `from core.test_runner import test_single_config, test_single_config_cli` |
| 构造函数签名 | `__init__(self, test_configs, parent=None)` | `__init__(self, config_manager, parent=None)` |
| 测试列表获取 | 外部传入 `test_configs` 列表 | 内部调 `self._cm.get_test_list()` |
| 重新测试 | 复用初始传入的列表 | 每次重新调 `get_test_list()`（可感知运行中新增/删除的配置） |

### 6. `main.py`（187 行，瘦身重写）

**行数变化**：484 → 187 行（减少 61%）

**删除的内容**（全部迁移到 `core/` 或 `ui/`）：
- 全局变量块（`APP_CONFIG`、`SETTINGS_DIR`、`CONFIGS` 等，第 27-32 行）
- 工具函数（`read_json`、`write_json`、`find_cfg_path`、`extract_website`、`derive_name`，第 35-73 行）
- 配置逻辑函数（`load_config`、`auto_discover_configs`、`detect_current`、`switch_config`，第 22-150 行）
- `create_icon()` 函数（第 153-168 行）
- `AddConfigDialog` 类（第 171-239 行）
- `CCSwitch.create_check_icon()` 方法（第 256-267 行）

**新增的导入**：
```python
from core.config_manager import ConfigManager, read_json
from ui.test_dialog import TestAllDialog
from ui.dialogs import AddConfigDialog
from ui.icons import create_tray_icon, create_check_icon
```

**CCSwitch 类改动对照**：

| 方法 | 重构前 | 重构后 |
|------|--------|--------|
| `__init__` | 调 `auto_discover_configs()` 全局函数 | `self.cm = ConfigManager()` + `self.cm.auto_discover()` |
| `build_menu` | 调 `detect_current(settings)`、`find_cfg_path()`、`read_json()` 全局函数 | 调 `self.cm.detect_current()`、`self.cm.find_cfg_path()`、`read_json()` |
| `build_menu` | `self.create_check_icon()` | `create_check_icon()`（模块函数） |
| `on_switch` | 调 `switch_config()` 全局函数 | 调 `self.cm.switch()` |
| `on_test_all` | 手动构建 test_list（20+ 行），传给 `TestAllDialog(test_list)` | `TestAllDialog(self.cm)` 一行 |
| `on_add_config` | 40+ 行内联业务逻辑（slug、冲突检测、文件写入、config 更新） | `self.cm.add(**values)` 一行 |
